## GraphRAG 판단 (GraphRAG 설계방법론 참조)

| # | 조건 | 충족 | 근거 |
|---|------|:----:|------|
| 1 | 동음이의어/용어 불일치 빈번 | ✓ | "면책"이 일반면책/특별면책/상법면책, 심사역마다 기준 해석 차이 |
| 2 | 인과관계 질의 >20% | ✓ | 4/14=29% — 특약→면책 관계, 할증 조건→보험료 |
| 3 | RAG 적중률 목표 미달 | ✓ | 약관 조항 간 참조 관계가 벡터만으로는 추적 불가 (면책 4조→특약 2조→할증 기준) |
| 4 | 동일 엔티티 다중 맥락 | ✓ | 같은 사고 유형이 약관(면책)/가이드라인(심사 기준)/과거 사례(판단 결과)에서 다르게 기술 |
| | **합계** | **4/4** | |

→ 결정: **승인** (4/4 만점, 모든 조건 충족)

### GraphRAG 구성 (pgvector + Apache AGE — 같은 PostgreSQL에서)
- **기술 스택**: pgvector + Apache AGE (PostgreSQL 16) — GraphRAG 설계방법론 §1 참조
- **노드**: 약관조항(ID, 조항번호, 유형), 사고유형(ID, 분류), 심사기준(ID, 상품별), 심사결과(approve/reject/defer)
- **엣지**: 참조(references: 면책4조→특약2조), 적용(applies_to: 사고유형→약관조항), 근거(based_on: 심사결과→심사기준), 동의어(synonym: 면책=면책사유=비보상)
- **활용**:
  - 질의 확장: "면책" 검색 시 관련 특약/할증 조건도 함께
  - 관계 추적: "이 사고 유형에 어떤 약관 조항이 적용되나?" → 그래프 탐색
  - 판단 근거: "과거 유사 사례에서 어떤 기준으로 판단했나?" → 사례→기준→조항 체인

## 검색 전략 (§5 참조)
- 검색 방식: **하이브리드 (벡터 + BM25)** — 약관은 조항 번호 키워드 매칭 중요
- 리랭킹: **적용** (§6) — 3개 컬렉션 크로스 검색 + 약관 복잡도로 노이즈 예상
- 크로스 컬렉션: GraphRAG로 약관 관계 추적 → `uw_guidelines` + `uw_cases` 동시 검색

### 검색 플로우
```
질의 입력
  ↓
① GraphRAG 질의 확장 (Apache AGE)
  - "면책" → "면책사유", "비보상", "특별면책" + 관련 조항 번호
  ↓
② 크로스 컬렉션 라우팅
  - "기준 뭐야?" → uw_guidelines
  - "유사 사례?" → uw_cases
  - "청구서 내용?" → claim_docs
  - 관계 질의 → uw_guidelines + uw_cases 동시
  ↓
③ 하이브리드 검색 (벡터 + BM25, RRF 결합)
  ↓
④ 리랭킹
  ↓
⑤ GraphRAG 컨텍스트 어셈블리
  - [관련 문서] + [관련 관계(약관→사고→판단)]
  ↓
⑥ LLM 응답 생성 + 판단 근거 포함
```

## 임베딩/모델 선택 (§4 참조)
- 임베딩: **Titan Embeddings V2** (Bedrock) — AWS 네이티브, 한국어 지원
