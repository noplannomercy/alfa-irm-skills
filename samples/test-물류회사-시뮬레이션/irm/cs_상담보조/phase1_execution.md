# CS 상담 보조 — 실행 배치 설계

> 작성일: 2026-03-26

## 실행(확정) — 결정적 파이프라인

### 반품 접수 파이프라인
```
Step 1: [접근] cs order --order-id → 주문 정보 확인
Step 2: [탐색] cs_knowledge에서 반품 절차 조회
Step 3: [접근] delivery track → 배송 완료 확인
Step 4: [판단] 규칙 기반 → 반품 가능 여부 (기간, 상태 등)
Step 5: [안내] 상담사에게 반품 가능/불가 + 절차 안내
```

- 입력: 주문번호
- 출력: 반품 가능 여부 + 절차 안내 텍스트
- 참고: 실제 반품 접수(시스템 기록)는 상담사가 직접 처리 — AI는 판단+안내만

### 개인정보 마스킹 파이프라인
```
Step 1: [입력] 텍스트 수신
Step 2: [패턴 매칭] 이름/전화번호/주소 정규식 탐지
Step 3: [마스킹] 탐지된 개인정보 치환 (홍길동 → 홍**)
Step 4: [출력] 마스킹된 텍스트 반환
```

- 입력: 원본 텍스트
- 출력: 마스킹된 텍스트
- 적용 시점: LLM 투입 전 자동 적용

## 실행(유연) — 판단 기반 프로세스

### 클레임 처리 안내 프로세스
```
Step 1: [접근] cs history → 해당 고객 상담 이력 조회
Step 2: [탐색] cs_knowledge에서 클레임 처리 가이드 검색
Step 3: [LLM 판단] 클레임 유형 판단 + 보상 기준 적용 제안
Step 4: [상담사 확인] 제안 검토 → 승인 또는 수정
```

- 판단 포인트: Step 3 — LLM이 클레임 유형 분류 + 기준 적용
- Human-in-the-loop: Step 4 — 상담사가 최종 결정
- 고정화 조건: 클레임 유형별 패턴이 안정되면 규칙 기반으로 전환

### 답변 초안 생성 프로세스
```
Step 1: [접근] 필요 시 배송 현황/주문 정보 조회
Step 2: [탐색] cs_knowledge에서 관련 FAQ/절차 검색
Step 3: [LLM] 검색 결과 + 문의 내용 기반으로 답변 초안 생성
Step 4: [상담사 확인] 초안 검토 → 전송 또는 수정
```

- 판단 포인트: Step 3 — LLM이 맥락에 맞는 답변 생성
- Human-in-the-loop: Step 4 — 상담사가 최종 답변 결정
