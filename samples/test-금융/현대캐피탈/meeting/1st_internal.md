## 기술 신호
- 거버넌스 무게: **Very Heavy** — 금감원 + 신용정보법 + 보조수단성 + 컴플라이언스
- GraphRAG 필요: **예 (강한 신호)** — PoC 실패 원인 "할부 vs 리스 규정 구분 못 함". 규정집이 상품별/상황별 분기가 복잡.
- 인프라 경로: **AWS Bedrock** (활성화 필요, CSAP 미확인)
- PoC 이력: **실패 (4원인 식별)** — 이 4가지 전부 해결해야 파일럿 성공

## PoC 실패 → 우리의 대응 매핑

| PoC 실패 원인 | 우리의 해결 방향 |
|-------------|--------------|
| 1. 2,000페이지 통째로 RAG → 관련 없는 규정 | 컬렉션 분리 + 메타데이터 필터링 (상품별/규정 유형별) |
| 2. 할부 vs 리스 규정 구분 못 함 | **GraphRAG — 상품-규정 관계 그래프** |
| 3. 심사역 반발 | UX 설계 + 점진적 도입 + 피드백 루프 |
| 4. 금감원 보고 방안 미수립 | 컴플라이언스 2차 미팅에서 사전 합의 |

## 신용정보 투입 불가 → AI 할 수 있는 것

```
AI가 할 수 있는 것 (신용정보 없이):
  ✓ 규정집에서 해당 규정 검색 (RAG)
  ✓ 내규/약관 적합성 확인 (규정 기반)
  ✓ 심사 의견서 초안 생성 (규정 근거 + 서류 정보)
  ✓ 유사 심사 사례 검색 (비식별화된 이력)

AI가 할 수 없는 것 (신용정보 필요):
  ✗ 신용 평가 / 신용 점수 기반 판단
  ✗ 연체 이력 기반 리스크 분석
  ✗ 소득 대비 상환 능력 산정
  → 이건 기존 CSS(자동 스코어링)가 이미 하고 있음
```

**핵심 메시지: "신용정보 판단은 CSS가, 규정/내규 검색은 AI가. 역할 분리."**

## 감지된 리스크
- **컴플라이언스 리스크**: 2차 미팅이 프로젝트 진행 가부를 결정. 규제 대응안 없이 가면 엎어짐.
- **정치적 리스크**: 오토에버가 PM인데 AI를 모름 → 커뮤니케이션 비용. 우리 산출물이 오토에버 경유로 전달되면 왜곡 가능.
- **심사역 수용성**: PoC 때 반발 이력 → "또 안 되겠지" 마인드. 파일럿 초기에 빠르게 "되네!" 경험 필요.
- **Bedrock CSAP**: Bedrock이 금융권 CSAP 인증 되어있는지 확인 필요. 안 되면 SageMaker + 로컬 모델로 전환.
