## 체이닝 패턴 (복합 질의)

- **"이 건 심사 의견서 초안 만들어줘"** (질의 #9) — 메인 체이닝
  = 접근(#5: CBS에서 심사 건 정보 — 신용정보 제외)
  → 탐색(#1: 여신규정에서 적용 규정 검색)
  → 탐색(#2: 자동차금융 특약에서 해당 조항)
  → 탐색(GraphRAG: 상품-규정 관계로 정확한 조항 특정)
  → 탐색(#4: 가이드라인에서 유사 사례)
  → 실행(유연: 규정 근거 + 사례 종합 → 의견서 초안)
  → Human-in-the-loop: 심사역 검토/수정/확정

- **"할부랑 리스 담보 기준이 다른 거 맞아?"** (질의 #10) — GraphRAG 핵심
  = 탐색(GraphRAG: "담보 평가" → 여신규정 vs 자동차특약 vs 리스특약 분기)
  → 탐색(#1,#2,#3: 각 규정의 담보 평가 조항 병렬 검색)
  → 응답: 상품별 차이점 비교

## Phase 0 결론

### 행위 유형 분포 요약
- **압도적 탐색 (86%)** — 규정집 검색이 거의 전부
- 접근/실행은 보조적 — CBS 조회 + 의견서 생성
- **아키텍처 시사**: RAG 품질이 곧 프로젝트 성패. GraphRAG는 PoC 실패 원인 해결의 핵심.

### GraphRAG 사전 신호
- 동음이의어/용어 불일치: **있음** — 같은 조항이 3개 규정에 각각 다르게 존재
- 인과관계 질의 비율: 2/14 (14%) — 경계선이지만...
- 동일 엔티티 다중 맥락: **있음** — "담보 평가"가 할부/리스/여신 3맥락
- **PoC 실패 원인이 곧 GraphRAG 부재** — 증거 기반 필수
- → 사전 판단: **필수 (PoC 실패 재현 방지)**
