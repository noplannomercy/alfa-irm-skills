# 불량원인분석_보조 — 실행 배치 설계

> 작성일: 2026-04-24
> 참조: IRM 실행가이드 v10

## 실행(확정) — 결정적 파이프라인

해당 없음. 이 유즈케이스의 모든 판단은 상황 의존적이며, 같은 불량코드라도 라인/시기/부품에 따라 원인이 다르다. 실행(확정)으로 고정할 수 있는 절차가 없음.

## 실행(유연) — 판단 기반 프로세스

### 프로세스 1: 불량 원인 분석 보조

현재 5단계 프로세스 중 **4단계(원인 분석)**에 해당.

```
Step 1: [접근] MES에서 불량 건 상세 조회 (cli-mes)
Step 2: [접근] 불량코드 변환 (cli-defectcode) — 타 공장 코드로 확장
Step 3: [탐색] 유사 불량 이력 검색 (defect_history 컬렉션 + GraphRAG 질의 확장)
Step 4: [탐색] 관련 검사기준서/작업표준서 검색 (quality_standards 컬렉션)
Step 5: [탐색] 유사 건의 과거 8D 보고서 참조 (investigation_reports 컬렉션)
Step 6: [LLM 판단] 수집된 정보 종합 → 원인 분석 + 추천 원인 제시
Step 7: [사람 확인] 품질 엔지니어가 추천 원인을 검토/수정/확정
```

- 판단 포인트: **Step 6** — LLM이 유사 이력 + 기준서 + 과거 대책을 종합하여 원인 후보를 제시
- Human-in-the-loop: **Step 7** — 품질 엔지니어가 최종 원인 확정
- 불일치 시 처리: AI 추천과 엔지니어 판단이 다를 경우, 엔지니어 판단이 우선. AI 추천과 최종 결정을 함께 기록 (피드백 루프)
- 고정화 조건: 동일 불량코드+동일 라인에서 AI 추천 원인이 5회 연속 엔지니어 확정과 일치 → 해당 패턴을 규칙화 검토 (실행(확정) 전환 후보)

### 프로세스 2: 8D 보고서 초안 생성

현재 5단계 프로세스 중 **5단계(보고서 작성)**에 해당.

```
Step 1: [입력] 프로세스 1의 결과 (확정된 원인 + 관련 이력/기준서)
Step 2: [탐색] 유사 건의 과거 8D 보고서 참조 (패턴/문체 학습)
Step 3: [LLM 판단] 8D 양식에 맞게 초안 생성
  - D1: 팀 구성 (기본값 자동 채움)
  - D2: 문제 정의 (불량 상세에서 자동 생성)
  - D3: 긴급 조치 (유사 건 참조)
  - D4: 근본 원인 (프로세스 1 결과)
  - D5: 시정 조치 (유사 건 대책 참조 + LLM 추천)
  - D6: 시정 조치 검증 (템플릿)
  - D7: 재발 방지 (LLM 추천)
  - D8: 팀 인정 (기본값)
Step 4: [사람 확인] 품질 엔지니어가 초안 검토/수정/승인
```

- 판단 포인트: **Step 3** — LLM이 양식별 내용 생성
- Human-in-the-loop: **Step 4** — 최종 승인은 반드시 사람
- 불일치 시 처리: 엔지니어가 크게 수정한 부분은 피드백으로 기록 → 프롬프트 개선에 활용
- 고정화 조건: D1(팀구성), D8(팀인정)은 이미 고정값. D2(문제정의)는 불량 상세에서 자동 생성 가능 → 부분 고정화 가능.
