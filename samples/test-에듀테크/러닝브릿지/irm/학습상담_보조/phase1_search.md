## 컬렉션 설계 (§1 참조)

### 분리 기준 판단

#### FAQ+규정 vs 강의 콘텐츠(교안+자막)
| 기준 | 판단 | 근거 |
|------|------|------|
| 갱신 주기 | 통합 가능 | 둘 다 거의 정적 (수시 vs 강의 등록 시) |
| 메타데이터 | **분리** | FAQ는 질문유형/카테고리, 교안/자막은 강의ID/카테고리/난이도/강사 — 스키마 겹치지 않음 |
| 질의 패턴 | **분리** | "수강 신청 방법?"(FAQ) vs "경사하강법 설명해줘"(교안/자막) — 거의 혼합 검색 없음 |
| **판단** | **2/3 → 분리** | |

#### 교안 vs 자막
| 기준 | 판단 | 근거 |
|------|------|------|
| 갱신 주기 | 통합 | 같은 강의 등록 시점에 함께 생성 |
| 메타데이터 | 통합 | 동일 (강의ID, 카테고리, 난이도) |
| 질의 패턴 | 통합 | "이 강의에서 X 설명해줘" — 교안과 자막이 함께 검색되어야 정확도 향상 |
| **판단** | **0/3 → 통합** | |

#### FAQ+규정 통합 여부
| 기준 | 판단 | 근거 |
|------|------|------|
| 갱신 주기 | 통합 | 둘 다 정적 |
| 메타데이터 | 약간 다름 | FAQ는 Q&A 유형, 규정은 조항번호. 하지만 doc_type으로 구분 가능 |
| 질의 패턴 | 약간 다름 | 하지만 둘 다 "상담 기준 문서"로 상담사가 함께 참조 |
| **판단** | **1/3 → 통합** | 규정이 10p (소규모)이므로 별도 컬렉션은 비효율 |

#### 상담 이력 vs 나머지
| 기준 | 판단 | 근거 |
|------|------|------|
| 갱신 주기 | **분리** | 실시간 (하루 500건) vs 정적 |
| 메타데이터 | **분리** | 상담일시/유형/상담사ID/수강생ID vs 강의ID/카테고리 |
| 질의 패턴 | **분리** | "비슷한 문의 전에 있었나?"(이력 검색) vs "수강 방법?"(기준 문서 검색) |
| **판단** | **3/3 → 분리** | |

### 컬렉션 구조

```
벡터 저장소 (pgvector)
├── 컬렉션: faq_and_rules
│   └── 포함: FAQ 150건 + 환불/변경 규정 (10p)
│   └── 용도: 수강 방법(30%) + 환불/변경(20%) + 기술 문제(15%) 질의 대응
│
├── 컬렉션: course_content
│   └── 포함: 강의 교안 PDF 3,000개 + 강의 자막 SRT 2,800개
│   └── 용도: 콘텐츠 질문(35%) 대응
│
└── 컬렉션: consultation_history
    └── 포함: Zendesk 상담 이력 36만건
    └── 용도: 유사 상담 패턴 검색, Few-shot 답변 참조
```

| 컬렉션명 | 포함 소스 | 메타데이터 | 인덱싱 범위 | 갱신 방식 |
|----------|----------|-----------|-----------|----------|
| faq_and_rules | FAQ 150건 + 환불규정 10p | doc_type, category, question_type | 전수 (최신형) | 수동/Webhook |
| course_content | 교안 3,000 + 자막 2,800 | course_id, category, level, instructor, content_type | 전수 (최신형) | 강의 등록/수정 시 |
| consultation_history | Zendesk 36만건 | ticket_date, inquiry_type, agent_id, resolution_status | 최근 1년 (이력형, 확장 가능) | 일배치 (Zendesk API) |

### 벡터 저장소 선택 (§1 제품 비교표 참조)
- **선택: pgvector** — 근거: 기존 RDS PostgreSQL 활용 가능, PoC 규모(소~중), 관리 단순, 추가 인프라 비용 없음
- **인덱스: HNSW** — 근거: 범용 최적, faq_and_rules는 소규모이므로 Flat도 가능하나 통일성 위해 HNSW

## 메타데이터 스키마 (§2 참조)

### 공통 메타데이터 (전 컬렉션)

```json
{
  "doc_id": "string",         // 문서 고유 ID
  "doc_type": "string",       // faq / rule / course_pdf / course_srt / ticket
  "source": "string",         // 원본 위치 (URL, S3 key, Zendesk ticket ID)
  "created_at": "date"        // 작성일/등록일
}
```

### faq_and_rules 도메인 메타데이터

```json
{
  "category": "string",           // 수강방법 / 환불변경 / 기술문제 / 기타
  "question_type": "string",      // faq_question / faq_answer / rule_clause
  "rule_section": "string|null"   // 환불규정 조항번호 (FAQ는 null)
}
```

### course_content 도메인 메타데이터

```json
{
  "course_id": "string",      // 강의 고유 ID
  "course_title": "string",   // 강의명
  "category": "string",       // 프로그래밍 / 데이터분석 / AI / 비즈니스 등
  "level": "string",          // 입문 / 초급 / 중급 / 고급
  "instructor": "string",     // 강사명
  "content_type": "string",   // pdf_page / srt_segment
  "chapter": "string|null"    // 챕터/섹션 번호
}
```

### consultation_history 도메인 메타데이터

```json
{
  "ticket_id": "string",          // Zendesk 티켓 ID
  "ticket_date": "date",          // 상담 일시
  "inquiry_type": "string",       // 수강방법 / 콘텐츠질문 / 환불변경 / 기술문제
  "resolution_status": "string",  // resolved / pending / escalated
  "agent_id": "string"            // 상담사 ID (비식별화)
}
```

### 추출 방식

| 메타데이터 | 추출 방식 | 근거 |
|-----------|----------|------|
| course_id, course_title, category, level | 소스 시스템(S3 경로/파일명) | 이미 구조화 |
| inquiry_type | LLM 기반 자동 추출 | Zendesk 티켓 본문에서 분류 |
| rule_section | 규칙 기반 (정규식) | "제N조" 패턴 |
| chapter | 규칙 기반 (PDF 목차/SRT 타임스탬프) | 구조적 패턴 |
| question_type | 규칙 기반 | FAQ는 Q/A 쌍, 규정은 조항 |

## 파싱 & 청킹 (§3 참조)

| 컬렉션 | 데이터 형태 | 도구 | 전략 |
|--------|-----------|------|------|
| faq_and_rules (FAQ) | 구글독스 (HTML/텍스트) | 커스텀 전처리 | 1 Q&A 쌍 = 1 청크. 질문+답변을 하나의 청크로 |
| faq_and_rules (규정) | PDF (10p) | Docling + HybridChunker | 조항 단위 구조 인식 청킹. merge_peers=True |
| course_content (교안) | PDF | Docling + HybridChunker | 섹션/페이지 단위 청킹. 테이블/이미지 설명 포함. PoC 시 50~100개로 범위 제한 |
| course_content (자막) | SRT | 커스텀 전처리 | 시간 구간별 병합 (5~10분 단위). course_id + chapter 메타 부착 |
| consultation_history | Zendesk 티켓 | 커스텀 전처리 | 1 티켓 = 1 청크. 질문+답변 포함. 수강생 식별정보 마스킹 |

### 데이터 품질 필터

| 소스 | 필터 기준 | 근거 |
|------|----------|------|
| 교안 PDF | 텍스트 추출 후 50자 미만 페이지 제외 | 이미지 전용 페이지는 의미 검색 불가 |
| 자막 SRT | 10초 미만 단독 세그먼트 제외 | 의미 없는 짧은 발화 |
| 상담 이력 | 상태=resolved만 인덱싱 | 미해결 건은 참조 품질 불확실 |
| 상담 이력 | 본문 20자 미만 제외 | "감사합니다" 등 무의미 응답 |

## 임베딩 모델 (§4 참조)

### 선정

| 기준 | 판단 |
|------|------|
| 언어 | 한국어 중심 (FAQ, 상담이력 전부 한국어) |
| 인프라 | AWS 기반, GPU 없음 |
| 단계 | PoC (빠른 시작 필요) |

- 청킹 시 토크나이저: PoC에서는 tiktoken (OpenAI), 전환 시 HuggingFace 토크나이저로 변경

## 검색 전략 (§5 참조)

### 컬렉션별 검색 방식

| 컬렉션 | 검색 방식 | top-k | 근거 |
|--------|----------|:-----:|------|
| faq_and_rules | 벡터 검색 단독 | 5 | 소규모(~160건), 키워드보다 의미 매칭 중심 |
| course_content | 하이브리드 (벡터+BM25) | 10 | 대규모(5,800+청크), 강의명/기술 용어 키워드 매칭 중요 |
| consultation_history | 하이브리드 (벡터+BM25) | 10 | 대규모, 특정 키워드("환불", "에러 코드") 정확 매칭 필요 |

### 결합 방식
- **RRF (Reciprocal Rank Fusion)** — 초기 가중치 튜닝 없이 안정적 시작. 모니터링 후 가중치 합산 전환 검토

### 메타데이터 필터링

```
질의: "파이썬 강의에서 변수 선언 어떻게 해요?"
→ 컬렉션: course_content
→ 필터: category = "프로그래밍" (추출 가능 시)
→ 또는 course_title LIKE "%파이썬%"

질의: "환불 규정에서 부분 환불 기준이 뭐예요?"
→ 컬렉션: faq_and_rules
→ 필터: doc_type = "rule_clause"
```

### 교차 컬렉션 검색

체이닝 질의(#7, #8)에서 탐색 단계:
```
"결제한 지 7일 됐는데 환불 가능한가요?"
→ 1단계: faq_and_rules 검색 (환불 규정 조항)
→ 2단계: 접근 레이어에서 수강생 결제일 조회 (PostgreSQL)
→ 3단계: 실행 레이어에서 규정 적용 판단
```
→ 교차 컬렉션 검색은 불필요. 라우팅 프롬프트가 질의를 분석하여 단일 컬렉션 선택.

### 라우팅 프롬프트 (컬렉션 선택)

```
질의 유형 → 대상 컬렉션:
- 수강 방법, 기술 문제, 환불/변경 → faq_and_rules
- 강의 내용 질문 → course_content
- 유사 문의 검색 → consultation_history
- 복합 질의 → 순차 라우팅 (체이닝)
```

## 리랭킹 (§6 참조)

### 적용 판단

| 컬렉션 | 적용 여부 | 근거 |
|--------|:--------:|------|
| faq_and_rules | 미적용 | 소규모(~160건), top-5로 충분 |
| course_content | **적용** | 대규모(5,800+청크), 하이브리드 검색 결과 통합 정렬 필요 |
| consultation_history | **적용** | 대규모(36만건), 관련 없는 티켓 필터링 필요 |

### 모델 선택
- **PoC**: FlashRank (CPU 동작, 프로토타이핑용, 무료)
- **본격 운영 시**: BGE-reranker-v2-m3 (로컬, 한국어 우수) 또는 Cohere Rerank 3.5 (API)

### 파이프라인
```
course_content / consultation_history:
→ 하이브리드 검색 top-20
→ FlashRank 리랭킹
→ 최종 top-5 → LLM 전달
```

## 갱신 파이프라인 (§8 참조)

### 갱신 긴급도

| 컬렉션 | 갱신 안 됐을 때 문제 | 긴급도 |
|--------|-------------------|:------:|
| faq_and_rules | 구버전 FAQ/규정이 히트 → 잘못된 안내 | **즉시** |
| course_content | 신규 강의가 검색 불가 → 불편하지만 위험하지 않음 | **허용** |
| consultation_history | 최신 상담이 누락 → 유사 사례 검색 불완전 | **허용** |

### 갱신 방식

| 컬렉션 | 갱신 방식 | 허용 지연 | 구현 |
|--------|----------|:--------:|------|
| faq_and_rules | 수동 트리거 | 즉시 (FAQ 변경 시 관리자 실행) | 삭제→재파싱→재임베딩 |
| course_content | 스케줄 배치 | 1일 | S3 이벤트 감지 → 일배치 처리 |
| consultation_history | 스케줄 배치 | 1일 | Zendesk API → 일배치 (전일 종결 티켓) |

### 증분 업데이트

```
문서 ID 체계:
  - FAQ: faq_{sequential_id}
  - 규정: rule_{section_number}
  - 교안: course_{course_id}_pdf_{page}
  - 자막: course_{course_id}_srt_{segment}
  - 티켓: zendesk_{ticket_id}

청크 ID: {collection}_{doc_id}_{chunk_index}
```

## GraphRAG 판단

| # | 조건 | 충족 | 근거 |
|---|------|:----:|------|
| 1 | 동음이의어/용어 불일치 빈번 | ✗ | 강의 콘텐츠는 독립적, 용어 혼동 없음 |
| 2 | 인과관계 질의 >20% | ✗ | 1/14 (7%) — #9 학습 추천만 해당 |
| 3 | RAG 적중률 목표 미달 | ✗ | 미측정 (PoC 후 평가) |
| 4 | 동일 엔티티 다중 맥락 | ✗ | 수강생-강의 관계는 DB 접근으로 해결 |
| | **합계** | **0/4** | |

→ 결정: **불필요** — 메타데이터 필터링(course_id, category) + per-course 인덱싱으로 대응. PoC 이후 적중률 평가 시 재검토.

## 설계 요약

```
┌─────────────────────────────────────────────────┐
```
