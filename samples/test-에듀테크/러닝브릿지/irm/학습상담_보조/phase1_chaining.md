## 체이닝 플로우 상세

### 패턴 A: 규정 적용 플로우 (질의 #7, #8)

공통 구조: **탐색 → 접근 → 실행(확정)**

```
사용자 질의
    │
    ▼
┌─────────────────────────────────────────────┐
│ ① 탐색: faq_and_rules 컬렉션               │
│    "환불 기준" / "변경 기준" 검색             │
│    → {rule_text, rule_section}              │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ ② 접근: cli-anything-student               │
│    enrollment list --student-id <ID>        │
│    → {payment_date, progress_pct, status}   │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ ③ 실행(확정): 규칙 기반 판단                │
│    judge_refund() / judge_course_change()   │
│    → {eligible, reason, rule_section}       │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ ④ 응답 생성                                │
│    "환불 가능합니다. 근거: 제N조 ..."        │
│    또는 "환불 불가합니다. 사유: ..."          │
└─────────────────────────────────────────────┘
```

**데이터 흐름:**
- ① → ②: 탐색 결과(규정 조항)는 ③에서 사용. ②는 ①과 독립적이므로 **병렬 실행 가능**.
- ①+② → ③: 규정 텍스트 + 수강생 데이터를 합쳐 판단
- ③ → ④: 판단 결과 + 근거 조항을 자연어로 변환

**에러 핸들링:**
- ① 실패 (규정 미발견): "해당 규정을 찾을 수 없습니다. 담당자 확인 필요." → 체이닝 중단
- ② 실패 (수강생 미발견): "수강생 정보 조회 실패. ID를 확인해주세요." → 체이닝 중단
- ③ 규칙 매칭 실패: "규정 범위 밖입니다." → 에스컬레이션 안내

### 패턴 B: 분석+추천 플로우 (질의 #9)

구조: **접근 → 실행(유연) → 탐색**

```
사용자 질의
    │
    ▼
┌─────────────────────────────────────────────┐
│ ① 접근: cli-anything-student               │
│    progress get --student-id <ID>           │
│    grade list --student-id <ID>             │
│    → {진도 데이터, 성적 데이터}              │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ ② 실행(유연): LLM 패턴 분석                │
│    "강점/약점 파악, 진도 지연 원인 추정"     │
│    → {strengths, weaknesses, analysis}      │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ ③ 탐색: course_content 컬렉션              │
│    LLM 분석 결과 기반 검색 쿼리 생성         │
│    필터: level, category                    │
│    → [{course_id, course_title, level}]     │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ ④ 실행(유연): LLM 추천 선정                │
│    패턴 분석 + 검색 결과 → 추천 + 사유       │
│    → [{recommended_course, reason}]         │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ ⑤ Human-in-the-loop                        │
│    상담사가 추천 결과 검토 → 수정/승인        │
│    → 최종 안내                               │
└─────────────────────────────────────────────┘
```

**데이터 흐름:**
- ① → ②: 접근 결과가 LLM 분석의 입력 (순차)
- ② → ③: LLM 분석 결과로 검색 쿼리 동적 생성 (순차)
- ③ → ④: 검색 결과를 LLM이 최종 선정 (순차)
- ④ → ⑤: 추천 결과를 상담사가 검토 (Human-in-the-loop)

**에러 핸들링:**
- ① 실패: "수강생 데이터 조회 실패" → 체이닝 중단
- ② LLM 분석 불확실: "확인 필요" 태그 → 상담사에게 직접 판단 요청
- ③ 검색 결과 없음: "조건에 맞는 강의가 없습니다" → 조건 완화 후 재검색
- ⑤ 상담사 거부: 추천 수정 → 피드백 기록 (고정화 판단 데이터)

## 체이닝 오케스트레이션

### LLM 라우팅 전략

```
질의 수신
    │
    ▼
┌──────────────────────────────┐
│ 라우팅 프롬프트               │
│ (질의 의도 분석 → 패턴 선택)  │
└──────┬───────────────────────┘
       │
  ┌────┼────────────────┐
  │    │                │
  ▼    ▼                ▼
 단순   패턴A            패턴B
 질의   (규정 적용)      (분석+추천)
  │    #7, #8           #9
  ▼    │                │
 탐색   탐색→접근→실행    접근→실행→탐색→실행
 only  (확정)           (유연, HitL)
```

### 라우팅 판단 기준

| 키워드/패턴 | → 라우팅 |
|-----------|---------|
| "환불", "변경", "취소" | 패턴 A (규정 적용) |
| "추천", "느린", "어떤 강의", "뭐 들으면" | 패턴 B (분석+추천) |
| 위 키워드 없음 | 단순 질의 (탐색 only) |

## 모니터링 지표

| 지표 | 측정 방법 | 목표 |
|------|----------|------|
| 체이닝 성공률 | 전체 완료 / 전체 시작 | >95% |
| 패턴 A 정확도 | 상담사 오버라이드 비율 | <5% (규칙이 정확해야 함) |
| 패턴 B 채택률 | 상담사 수정 없이 채택 / 전체 | >60% (초기), >80% (학습 후) |
| 평균 체이닝 시간 | 질의→응답 전체 | 패턴 A: <3초, 패턴 B: <10초 |
| 고정화 후보 | 동일 판단 연속 횟수 | 10회 이상 → 규칙화 검토 알림 |
