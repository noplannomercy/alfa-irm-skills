## 1. 컬렉션 설계 (§1)

### 1-1. 컬렉션 분리 판단 (3기준 각각)

Phase 0 분류 결과를 바탕으로 3개 컬렉션으로 확정한다.

#### 컬렉션 1: `safety_knowledge` (R-01+R-02+R-05 통합)

| 분리 기준 | 판단 | 근거 |
|---------|------|------|
| 갱신 주기 | **통합** | 매뉴얼/SOP/체크리스트 모두 6개월~1년 유사, 별도 파이프라인 불필요 |
| 메타데이터 | **통합** | 공종(specialty), 작업유형(work_type) — 통일 스키마 적용 가능 |
| 질의 패턴 | **통합** | "기준 알려줘", "절차가 어떻게 돼?" — 함께 검색 시 풍부한 답변, 함께 조회됨 |
| **판단** | **통합** | 0/3 분리 기준 해당 → 통합 타당 |

#### 컬렉션 2: `regulation` (R-03 단독)

| 분리 기준 | 판단 | 근거 |
|---------|------|------|
| 갱신 주기 | **분리** | 법 개정 시 즉시 갱신 (연 2~3회), 매뉴얼과 파이프라인 다름. 지연 시 법 위반 위험 |
| 메타데이터 | **분리** | 법령명, 조항번호, 시행일 등 법규 고유 메타 → 매뉴얼 스키마와 호환 불가 |
| 질의 패턴 | **분리** | "법적으로 어떻게 규정하나?" — 법규만 히트해야 정밀도 높음. 매뉴얼 혼재 시 노이즈 |
| **판단** | **분리** | **3/3 해당 → 분리 확정** |

#### 컬렉션 3: `accident_cases` (R-04 단독)

| 분리 기준 | 판단 | 근거 |
|---------|------|------|
| 갱신 주기 | **분리** | 사고 발생 시마다 추가 (이력형), 매뉴얼은 정적 문서 (최신형). 파이프라인 완전히 다름 |
| 메타데이터 | **분리** | 사고유형, 심각도, 원인분류, 연도 등 이력형 고유 메타 → 매뉴얼 스키마와 호환 불가 |
| 질의 패턴 | **분리** | "TBM 사례 찾아줘" — 사고 이력만 히트해야 함. 매뉴얼 혼재 시 관련성 하락 |
| **판단** | **분리** | **3/3 해당 → 분리 확정** |

### 1-2. 최종 컬렉션 구조

```
PostgreSQL (AWS RDS)
벡터 확장: pgvector
그래프 확장: Apache AGE

pgvector 컬렉션 (테이블):
├── safety_knowledge     → 매뉴얼(R-01) + SOP(R-02) + 체크리스트(R-05)
├── regulation           → 법규집(R-03) [최신형, 즉시 갱신]
└── accident_cases       → 사고 사례집(R-04) [이력형, 5년치]

Apache AGE 그래프:
└── safety_graph         → 동의어, 법규-문서 관계, 사고 인과 관계

실시간 조회 (인덱싱 없음):
├── R-07 safety_mgmt DB  → PostgreSQL 직접 쿼리
├── R-08 현장관리 API    → REST 호출
└── R-09 기상청 API      → REST 호출
```

### 1-3. 벡터 DB 선정: pgvector

| 판단 기준 | 근거 |
|---------|------|
| 기존 인프라 활용 | AWS RDS PostgreSQL 이미 운영 중 → 동일 인스턴스에 확장만 추가 |
| 관리 단순성 | IT팀 3명, 50% 가용 → 새로운 DB 운영 부담 최소화 |
| Apache AGE 통합 | pgvector + Apache AGE 동일 PostgreSQL에서 운영 (MS 표준 GraphRAG 스택) |
| 규모 적합성 | 예상 청크 수: safety_knowledge ~5,000청크, regulation ~8,000청크, accident_cases ~2,000청크 → 소~중규모 |

### 1-4. 인덱스 타입: HNSW

모든 컬렉션에 HNSW 적용. 규모(최대 15,000청크)에서 범용 최적.

## 2. 컬렉션별 메타데이터 스키마 (§2)

### 공통 메타데이터

```json
{
  "doc_type": "string",      // "manual" / "sop" / "checklist" / "regulation" / "accident"
  "work_type": "string",     // "고소작업" / "굴착" / "크레인" / "용접" / "전기" / "all"
  "date": "date",
  "source_file": "string"    // 원본 파일명
}
```

### safety_knowledge 도메인 메타

```json
{
  "specialty": "string",     // "토목" / "건축" / "설비" / "전기" / "공통"
  "doc_category": "string",  // "manual" / "sop" / "checklist"
  "chapter": "string"        // 규칙 기반 추출 (헤딩 텍스트)
}
```

### regulation 도메인 메타

```json
{
  "law_name": "string",          // "산업안전보건법" / "시행령" / "시행규칙" / "고시"
  "article_no": "string",        // "제44조" (규칙 기반 추출)
  "effective_date": "date",      // 시행일
  "regulation_type": "string"    // "osh_act" / "serious_accident_act"
}
```

### accident_cases 도메인 메타

```json
{
  "accident_type": "string",     // "추락" / "협착" / "감전" / "화재" / "폭발" / "기타"
  "severity": "string",          // "사망" / "중상" / "경상"
  "root_cause": "string",        // LLM 추출: "안전수칙 미준수" / "장비 결함" / "기상 악화" 등
  "year": "integer"              // 발생 연도 (2021~2025)
}
```

## 3. 문서 파싱 및 청킹 (§3)

### 3-1. 파싱 도구: Docling + HybridChunker (권장)

| 컬렉션 | 데이터 형태 | 파싱 도구 | 청킹 전략 |
|--------|-----------|---------|---------|
| safety_knowledge (매뉴얼, SOP) | PDF (구조 있음) | Docling + HybridChunker | 섹션/조항 단위 구조 인식, merge_peers=True |
| safety_knowledge (체크리스트) | Excel | 커스텀 전처리 | 1 항목 그룹 = 1 청크 (공종별) |
| regulation (법규집) | PDF (법 조항 구조) | Docling + HybridChunker | 조항 단위 청킹 (제○조 = 청크 경계) |
| accident_cases (Excel) | Excel | 커스텀 전처리 | 1 사고 = 1 청크 (사고유형+원인+조치+재발방지 통합) |
| accident_cases (PDF 상세) | PDF | Docling + HybridChunker | 보고서 섹션 단위 |

### 3-2. 데이터 품질 필터 기준 — 스캔 도면 (R-06, 1차 제외)

> R-06은 1차 PoC에서 제외하나, 2차 적용 시 아래 기준을 적용한다.

| 필터 기준 | 기준값 | 측정 방법 | 미충족 시 처리 |
|---------|--------|---------|------------|
| 최소 추출 텍스트 | **500자 이상** | OCR 후 텍스트 길이 카운트 | 인덱싱 제외, 품질 불량 로그 기록 |
| OCR 신뢰도 | **70% 이상** | Tesseract confidence score 평균 | 70% 미만 → 인덱싱 제외 |
| 이미지 해상도 | **150 DPI 이상** | PDF 메타데이터 또는 이미지 분석 | 미달 → OCR 전 해상도 업스케일 시도 |
| OCR 도구 | **Tesseract v5** (로컬, 도면 외부 전송 제한 고려) | - | AWS Textract는 외부 전송 허용 시만 사용 |
| 한국어 인식 | **kor+eng 언어팩** | Tesseract --lang kor+eng | - |

| 컬렉션 | 결합 방식 | top-k | 근거 |
|--------|---------|-------|------|
| safety_knowledge | RRF | 7 | 중규모, 초기에는 RRF로 시작 |
| regulation | RRF | 5 | 법 조항 검색 특성상 키워드 강함 |

### 5-2. 교차 컬렉션 검색 (Q-06 복합 질의)

```
Q-06 질의 → 질의 분류기
    → safety_knowledge 검색 (top-7)
    → regulation 검색 (top-5)
    → accident_cases 검색 (top-5)
    → 전체 17건 통합 후 리랭킹 1회
    → 최종 top-5 → LLM 전달
```

## 9. GraphRAG 판단 (§9~§12 참조)

### GraphRAG 적용 결정: ✅ 적용

| 판단 기준 | 충족 여부 | 근거 |
|---------|---------|------|
| 동의어 처리 필요 | ✅ | "고소작업=높이작업" (고객 발언), 건설 현장 용어 불일치 |
| 계층 관계 존재 | ✅ | 법규 → 매뉴얼 → SOP → 점검항목 계층 |
| 인과 관계 추적 | ✅ | 사고원인 → 위험요인 → 재발방지 대책 연결 |
| 멀티홉 탐색 가치 | ✅ | "크레인 관련 법규+사고원인+재발방지 체인" |

### GraphRAG 핵심 엔티티/관계

```
엔티티 유형:
  - 작업유형 (고소작업, 굴착, 크레인, 용접...)
  - 법규 (산업안전보건법 제44조, 중대재해처벌법...)
  - 위험요인 (추락, 협착, 감전, 붕괴...)
  - 안전조치 (안전대, 안전모, 추락방지망...)
  - 사고사례 ID

관계 유형:
  - 동의어 (고소작업 ←→ 높이작업)
  - 적용법규 (고소작업 → 산업안전보건법 제44조)
  - 구체화 (법 조항 → 매뉴얼 섹션)
  - 원인 (안전수칙 미준수 → 추락 사고)
  - 예방 (안전대 착용 → 추락 예방)
```

### 기술 스택: pgvector + Apache AGE (동일 PostgreSQL)

> Microsoft Azure Database for PostgreSQL GraphRAG 공식 표준 (GA 2025)

```sql
-- 초기화
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS age;
LOAD 'age';
SET search_path = ag_catalog, "$user", public;

-- 그래프 공간 생성
SELECT * FROM ag_catalog.create_graph('safety_graph');
```

### 탐색 깊이: 2홉 (권장)

```
질의: "고소작업 안전 기준"
1홉: 고소작업 → [동의어] → 높이작업
     고소작업 → [적용법규] → 산업안전보건법 제44조
2홉: 산업안전보건법 제44조 → [구체화] → 안전관리 매뉴얼 3장
     높이작업 → [관련사고] → 사고사례 #023
```
