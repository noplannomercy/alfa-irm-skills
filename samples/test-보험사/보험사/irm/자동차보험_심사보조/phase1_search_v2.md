## 컬렉션 설계 (§1 참조)

### 분리 기준 판단

#### 가이드라인+약관+FAQ+절차서(#1,2,3,7,8) vs OCR 청구서(#6)

| 기준 | 판단 | 근거 |
|------|------|------|
| 갱신 주기 | **분리** | 문서류=수동(연/월), OCR=배치(접수 시 실시간). 갱신 파이프라인 완전히 다름 |
| 메타데이터 | **분리** | 문서류=doc_type/product/section, OCR=claim_type/date/status. 스키마 호환 불가 |
| 질의 패턴 | **분리** | "심사 기준 뭐야?" vs "이 청구서 내용 확인" — 거의 섞이지 않음, 분리 시 정밀도 향상 |
| **판단** | **3/3 → 분리** | |

#### 가이드라인+약관+FAQ+절차서(#1,2,3,7,8) — 내부 통합 vs 분리?

| 기준 | 판단 | 근거 |
|------|------|------|
| 갱신 주기 | **통합** | 모두 수동, 연/월 유사 |
| 메타데이터 | **통합** | doc_type, product, section — 통일 스키마 적용 가능 |
| 질의 패턴 | **통합** | "기준 뭐야?", "절차 어떻게?" — 함께 검색 시 약관+가이드라인+절차가 동시에 나와야 풍부한 답변 |
| **판단** | **0/3 → 통합** | |

#### 심사 이력(#4)은 접근 유형 — 인덱싱 대상 검토

심사 이력 DB는 Phase 0에서 "접근"으로 분류됨 (Q1=Y, 위치 알고 있음 → PostgreSQL 직접 조회). 그러나 "유사 사례 찾아줘" 질의(#1, #9)는 의미 기반 검색이 필요하므로, **심사 이력 중 심사 의견 텍스트**는 벡터 인덱싱 대상.

→ 심사 이력의 **텍스트 필드(심사 의견, 판단 근거)**만 추출하여 별도 컬렉션으로 인덱싱.

| 기준 | 판단 | 근거 |
|------|------|------|
| 갱신 주기 | **분리** | 심사 완료 시마다 추가 (이력형), 문서류는 정적 (최신형) |
| 메타데이터 | **분리** | case_type/decision/reviewer/accident_type — 문서류와 완전 다름 |
| 질의 패턴 | **분리** | "유사 사례 찾아줘" — 사례만 히트해야 함 |
| **판단** | **3/3 → 분리** | |

### 최종 컬렉션 구조

```
pgvector (기존 PostgreSQL RDS 확장)
├── uw_guidelines        ← 가이드라인(#1) + 약관(#2) + FAQ(#3) + 절차서(#7,#8)
│   유형: 최신형 (구버전 삭제)
│   규모: ~500p + 200항목 ≈ 예상 5,000~8,000 청크
│
├── uw_cases             ← 심사 이력(#4) 중 텍스트 필드 추출
│   유형: 이력형 (축적)
│   규모: 자동차 5년 27만건 중 심사 의견 텍스트
│
└── claim_docs           ← OCR 청구서(#6)
    유형: 이력형 (축적)
    규모: 2년 16만건 (품질 필터 적용 후)

Apache AGE (같은 PostgreSQL)
└── uw_graph             ← 약관 관계 + 동의어 + 사고유형-심사결정 관계
```

### 벡터 저장소 선택 (§1 제품 비교표)

| 판단 기준 | 근거 |
|---------|------|
| 기존 인프라 | PostgreSQL RDS 이미 운영 중 → 확장만 추가 |
| 규모 | 예상 총 청크 수 소~중규모 → pgvector 적합 |
| GraphRAG 통합 | pgvector + Apache AGE 동일 PostgreSQL (MS 공식 표준) |
| IT팀 부담 | 4명, 별도 DB 운영 최소화 |

→ **pgvector** (PostgreSQL 확장), 인덱스: **HNSW**

## 메타데이터 스키마 (§2 참조)

### 공통 메타데이터
```json
{
  "doc_type": "string",    // guideline/terms/faq/procedure/case/claim
  "product": "string",     // auto/fire/liability/accident
  "source_file": "string", // 원본 파일명
  "date": "date"           // 작성/발행일
}
```

### uw_guidelines 도메인 메타데이터
```json
{
  "section": "string",       // 챕터/섹션명
  "article_no": "string",   // 약관 조항번호 (해당 시)
  "doc_category": "string"  // guideline/terms/faq/procedure
}
```

### uw_cases 도메인 메타데이터
```json
{
  "case_type": "string",     // 사고 유형
  "decision": "string",      // approve/reject/defer
  "reviewer": "string",      // 심사역 ID (가명)
  "accident_type": "string", // 상세 사고 분류
  "year": "integer"          // 심사 연도
}
```

### claim_docs 도메인 메타데이터
```json
{
  "claim_type": "string",  // 사고 유형
  "claim_date": "date",    // 접수일
  "status": "string"       // 처리 상태
}
```

## GraphRAG 판단

| # | 조건 | 충족 | 근거 |
|---|------|:----:|------|
| 1 | 동음이의어/용어 불일치 | ✓ | "면책"이 일반면책/특별면책/상법면책, 심사역마다 해석 차이 |
| 2 | 인과관계 질의 >20% | ✓ | 4/14=29% — 특약→면책, 할증→보험료, 면책→특약 관계 |
| 3 | RAG 적중률 목표 미달 | ✓ | 약관 조항 간 참조 관계가 벡터만으로는 추적 불가 |
| 4 | 동일 엔티티 다중 맥락 | ✓ | 같은 사고 유형이 약관/가이드라인/과거 사례에서 다르게 기술 |
| | **합계** | **4/4** | |

→ 결정: **승인 (4/4 만점)**

## GraphRAG 구성 (§9~§12 참조)

- **기술 스택**: pgvector + Apache AGE (같은 PostgreSQL 16, MS 공식 표준)
- **노드**: 약관조항(article_id, 조항번호, 유형), 사고유형(type_id, 분류), 심사기준(criterion_id, 상품별), 심사결과(decision)
- **엣지**: references(조항→조항), applies_to(사고유형→약관), based_on(심사결과→심사기준), synonym(면책=면책사유=비보상)
- **활용**:
  - 질의 확장: "면책" → 관련 특약/할증 조건 자동 확장
  - 관계 추적: "이 사고 유형에 어떤 약관 조항?" → 그래프 2-hop 탐색
  - 판단 근거: 사례→기준→조항 체인 추적

| 필터 | 기준 | 근거 |
