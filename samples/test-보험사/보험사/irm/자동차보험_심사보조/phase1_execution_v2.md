# 자동차보험 심사 보조 — 실행 배치 설계 (/irm-exec 단독)

> 작성일: 2026-03-27
> 실행 질의: 4/14건 (확정 2 + 유연 2 = 28%)
> 설계 신뢰도: ✅ (절차서 실제 확인)
> 참조: IRM 실행가이드 v10

---

## 실행(확정) — 결정적 파이프라인

### 가명처리 파이프라인

```
Step 1: 텍스트 수신 (심사 이력, 청구서 등)
Step 2: 패턴 매칭:
         - 이름 (한글 2~4자) → 홍*동
         - 주민번호 (6-7자리) → 완전 제거
         - 주소 → 시/구까지만
         - 연락처 → 010-****-5678
Step 3: 가명 텍스트 반환, 원본 미저장
```

- 입력: 원본 텍스트
- 출력: 가명처리된 텍스트
- 적용 시점: LLM 투입 전 자동 (모든 탐색/실행 파이프라인 앞단)
- 성공/실패: 미탐지 → 로그 + 알림

### 감사로그 파이프라인

```
Step 1: AI 보조 요청 수신
Step 2: AI 응답 생성 완료
Step 3: 로그 기록:
         - 요청 내용 (가명처리 후)
         - AI 응답 (검색 결과 참조 목록 + 판단 근거)
         - 심사역 최종 판단 (승인/거절/보류)
         - 심사역 수정 사항 (AI 의견과 다를 경우)
         - 타임스탬프 + 심사역 ID
Step 4: 심사 DB 감사로그 테이블 저장
```

- 입력: AI 요청+응답+심사역 판단
- 출력: 감사로그 레코드
- 금감원 대비: AI 판단 근거 + 심사역 최종 판단 함께 보관 (5년+)
- 성공/실패: 로그 누락 → 즉시 알림 (누락 0건 목표)

### 재심사 절차 파이프라인 (#8 절차서 기반 ✅)

```
Step 1: [접근] uw case → 원심사 건 조회
Step 2: [탐색] uw_cases → 유사 재심사 사례 검색
Step 3: [탐색] uw_guidelines → 재심사 기준 조회
Step 4: [판단] 규칙 기반 → 재심사 접수 가능 여부
         - 거절 후 30일 이내?
         - 새로운 증빙 있음?
         - 이의 사유 유효?
Step 5: [안내] 재심사 가능/불가 + 3단계 절차 안내
```

- 입력: 원심사 case_id
- 출력: 재심사 가능 여부 + 절차 안내
- 결정적: 같은 조건 → 같은 결과

## 실행(유연) — 판단 기반 프로세스

### 심사 의견 초안 생성 (#7 절차서 기반 ✅)

```
Step 1: [접근] core contract → 계약 정보
Step 2: [접근] uw history → 과거 심사 이력
Step 3: [탐색] uw_guidelines → 약관 조항 + 심사 기준 (GraphRAG 관계 추적)
Step 4: [탐색] uw_cases → 유사 사례 (GraphRAG 질의 확장)
Step 5: [LLM 판단]
         - 적용 약관 조항 정리
         - 유사 사례 판단 결과 참조
         - 심사 의견 초안 생성
         - 판단 근거 명시 (출처 포함)
Step 6: [심사역 확인] 초안 검토 → 승인/수정/거절
Step 7: [감사로그] AI 의견 + 심사역 최종 판단 기록
```

- 판단 포인트: Step 5 — LLM이 약관 적용 + 유사 사례 참조 + 의견 초안
- Human-in-the-loop: Step 6 — 심사역 최종 판단 (보조수단성 충족)
- 고정화 조건: 단순 심사(60%)에서 동일 판단 **5회 연속** → 규칙화 검토 알림

### 할증/특약 적용 판단 보조

```
Step 1: [접근] core contract → 계약/보험료 정보
Step 2: [접근] core premium → 현재 할증/할인 상태
Step 3: [탐색] uw_guidelines → 할증 기준 (GraphRAG: 할증→사고유형→약관 관계)
Step 4: [LLM 판단] 기준 대비 적용 여부 + 근거
Step 5: [심사역 확인] 판단 결과 검토
```

- 판단 포인트: Step 4
- Human-in-the-loop: Step 5
- 고정화 조건: 동일 할증 유형 **3회 연속** 동일 판단 → 규칙화 검토

## 체이닝 설계

### 체이닝 패턴

| # | 질의 | 체이닝 순서 | GraphRAG | 신뢰도 |
|---|------|-----------|:--------:|:------:|
| 1 | "이 건 심사해줘" | 접근→탐색→탐색→실행(유연)→실행(확정) | Y | ✅ |
| 2 | "이 특약 적용 가능?" | 탐색→접근→실행(유연) | Y | ✅ |
| 3 | "거절 건 재심사" | 접근→탐색→실행(확정) | N | ✅ |

### 패턴 1: 전체 심사 보조 (가장 복잡)

```
"이 건 심사해줘" →
  ① 접근: core contract → 계약 정보 [uw CLI]
  ② 접근: uw history → 과거 심사 이력 [uw CLI]
  ③ 탐색: uw_guidelines → 약관+기준 [GraphRAG 관계 추적]
  ④ 탐색: uw_cases → 유사 사례 [GraphRAG 질의 확장]
  ⑤ 실행(유연): LLM 의견 초안 + 판단 근거 → 심사역 확인
  ⑥ 실행(확정): 감사로그 기록
```

### 패턴 2: 특약/할증 판단

```
"이 특약 적용 가능?" →
  ① 탐색: uw_guidelines → 약관 조항 [GraphRAG: 특약→면책→할증]
  ② 접근: core contract → 계약 정보 [core CLI]
  ③ 실행(유연): LLM 적용 판단 + 근거 → 심사역 확인
```

### 패턴 3: 재심사

```
"거절 건 재심사" →
  ① 접근: uw case → 원심사 건 [uw CLI]
  ② 탐색: uw_cases + uw_guidelines → 유사 재심사 사례 + 기준
  ③ 실행(확정): 재심사 접수 가능 판단 + 절차 안내
```

---

## 주의 사항

- 이 설계는 **실행(확정/유연) + 체이닝만** 다룸
- 탐색(컬렉션/RAG)은 `/irm-search`, 접근(CLI)은 `/irm-access` 참조
- 체이닝의 ①② 접근/탐색 단계는 해당 스킬 결과를 참조
- 모든 실행(유연)에 Human-in-the-loop 필수 (보조수단성)
- 감사로그는 모든 체이닝 패턴의 마지막에 자동 실행
