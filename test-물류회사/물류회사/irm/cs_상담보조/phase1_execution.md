# CS 상담 보조 — 실행 배치 설계

> 작성일: 2026-04-15
> 설계 신뢰도: ✅ (절차서/가이드 실제 확인)

## 실행(확정) — 결정적 파이프라인

### 반품 판단 파이프라인 (✅ 절차서 기반)
```
Step 1: [접근] cs order --order-id → 주문 정보 (주문일, 상품)
Step 2: [접근] delivery track → 배송 상태 (수령 확인)
Step 3: [탐색] cs_knowledge → 반품 절차서 조회
Step 4: [판단] 규칙 기반:
         - 수령 후 7일 이내? → Y/N
         - 미개봉? → Y/N (상담사 확인)
         - 파손 여부? → Y/N (상담사 확인)
Step 5: [안내] 반품 가능/불가 + 절차 안내
```
- 입력: 주문번호
- 출력: 반품 가능 여부 + 5단계 절차 안내
- 결정적: 같은 조건(7일/미개봉/미파손) → 항상 같은 결과

### 개인정보 마스킹 파이프라인 (✅ 정책 기반)
```
Step 1: 텍스트 수신
Step 2: 패턴 매칭:
         - 이름 (한글 2~4자) → 홍*동
         - 전화번호 (010-XXXX-XXXX) → 010-****-5678
         - 주소 → 시/구까지만
Step 3: 마스킹 적용
Step 4: 마스킹 텍스트 반환 + 원본 미저장
```
- 적용 시점: LLM 투입 전 자동

## 실행(유연) — 판단 기반 프로세스

### 클레임 처리 안내 (✅ 가이드 기반)
```
Step 1: [접근] cs history → 상담 이력 조회
Step 2: [탐색] cs_knowledge → 클레임 가이드 검색
Step 3: [LLM 판단] 유형 분류 + 보상 기준 적용:
         - 파손: 사진 확인 → 100% 환불 또는 교환
         - 분실: 택배사 확인 → 보상
         - 지연: 기간별 쿠폰/환불 차등
         - 오배송: 즉시 회수 + 재발송
Step 4: [상담사 확인] 제안 검토 → 승인 또는 수정
```
- 판단 포인트: Step 3 — LLM이 유형 판단 + 기준 적용
- Human-in-the-loop: Step 4
- 고정화 조건: 동일 유형 동일 판단 3회 연속 시 규칙화 검토

### 답변 초안 생성
```
Step 1: [접근] 필요 시 배송/주문 조회
Step 2: [탐색] cs_knowledge → 관련 FAQ/절차
Step 3: [LLM] 검색 결과 + 문의 맥락 → 답변 초안
Step 4: [상담사 확인] 검토 → 전송 또는 수정
```
