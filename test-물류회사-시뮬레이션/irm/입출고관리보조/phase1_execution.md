# 입출고 관리 보조 — 실행 배치 설계

> 작성일: 2026-05-15

## 실행(확정) — 결정적 파이프라인

### 검수 판단 파이프라인
```
Step 1: [접근] wms inbound → 입고 품목/수량 확인
Step 2: [탐색] logistics_docs에서 해당 품목 검수 기준 조회
Step 3: [판단] 규칙 기반 → 기준 대비 합격/불합격/보류
Step 4: [안내] 담당자에게 판단 결과 + 근거 제시
```
- 입력: 바코드 스캔 결과 (품목코드)
- 출력: 합격/불합격/보류 + 기준 근거
- 결정적: 같은 품목 + 같은 기준 = 같은 결과

### 이상 건 기록 파이프라인
```
Step 1: [입력] 이상 유형, 품목, 설명 수신
Step 2: [GraphRAG] 용어 표준화 (자유 텍스트 → 표준 이상 유형)
Step 3: [접근] wms defect-record → WMS에 기록
Step 4: [확인] 기록 완료 확인 반환
```
- 입력: 이상 건 내용 (자유 텍스트)
- 출력: 표준화된 기록 + WMS 저장 확인
- 핵심: Step 2에서 GraphRAG가 "깨짐" → "파손(DAMAGE)" 표준화

## 실행(유연) — 판단 기반 프로세스

### 이상 건 처리 판단 프로세스
```
Step 1: [접근] wms defect-list → 해당 이상 건 상세
Step 2: [탐색] defect_history에서 유사 건 검색 (GraphRAG 질의 확장)
Step 3: [탐색] logistics_docs에서 이상 처리 절차 조회
Step 4: [LLM 판단] 유형 판단 + 유사 건 참조 + 처리 방법 제안
Step 5: [담당자 확인] 제안 검토 → 승인 또는 수정
```
- 판단 포인트: Step 4 — LLM이 유형 분류 + 처리 방법 제안
- Human-in-the-loop: Step 5 — 담당자 최종 결정
- 고정화 조건: 동일 유형 처리가 3회 연속 같은 결과면 규칙화 검토

### 원인 분석 프로세스
```
Step 1: [접근] wms defect-stats → 기간별 이상 건 통계
Step 2: [탐색] defect_history에서 패턴 검색 (품목별, 센터별, 시기별)
Step 3: [LLM 분석] 통계 + 이력 기반 원인 추론
Step 4: [담당자 확인] 분석 결과 검토
```
- 판단 포인트: Step 3 — LLM이 패턴 기반 원인 추론
- Human-in-the-loop: Step 4 — 물류운영팀장 검토
