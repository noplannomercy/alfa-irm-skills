## ALFA 스킬 (고객 대응)

### /alfa-prep — 미팅 전 준비

| 항목 | 내용 |
|------|------|
| **언제** | 고객 미팅 전 |
| **입력** | 고객 정보 (회사명, 업종, 규모, 상황 — 아는 만큼) |
| **출력** | `prep_질문리스트.md`, `prep_사전요청서.md`, `profile.md`, `STATUS.md` |
| **핵심** | 프로필 6항목 매핑 → 빈칸 기반 질문 생성 + 업종별 분기 |

**프로필 6항목:** 회사 개요 / AI현황 / 인프라 / 규제 / 원하는 것 / 제약사항

**업종 분기:**
- 금융 → 규제 Heavy (금감원, 컴플라이언스팀)
- 제조 → 온프레미스 분기 (데이터 반출, GPU)
- IT서비스 → 경량 거버넌스
- 이커머스 → 2단계 접근 (내부→외부)
- 의료/공공 → 규제 Very Heavy

### /alfa-live — 미팅 중/후 프로필 업데이트

| 항목 | 내용 |
|------|------|
| **언제** | 미팅 직후 |
| **입력** | 미팅 메모 (자유 형식 — 메모, 녹음 텍스트 등) |
| **출력** | `Nst_script.md`, profile 업데이트, STATUS 업데이트 |
| **핵심** | 프로필 업데이트 + 신호 감지 + 미확인 항목 알림 |

**자동 감지 신호 6가지:**

| 신호 | 고객 발언 | 태그 |
|------|----------|------|
| GraphRAG 필요 | "같은 건데 다르게 불러요" | #GraphRAG |
| 온프레미스 | "외부 반출 금지" | #on-premise |
| 거버넌스 격상 | "금감원", "고객 정보" | #gov-heavy |
| 2단계 접근 | "고객 챗봇도 하고 싶고" | #2-phase |
| 스폰서 미확정 | "위에서 아직" | #sponsor-pending |
| PoC 실패 이력 | "전에 해봤는데" | #poc-history |

**여러 번 호출 가능** — 같은 미팅 추가 메모도, 새 미팅도 가능.

### /alfa-wrap — 미팅 후 산출물 생성

| 항목 | 내용 |
|------|------|
| **언제** | 미팅 끝나고 (미팅마다 또는 전부 끝나고) |
| **입력** | 없음 (기존 폴더에서 자동 로드) |
| **출력** | 아래 표 참조 (조건부 생성) |
| **핵심** | summary 없는 회차에 대해 자동 생성 + 프로필 완성도에 따라 Phase 문서/유즈케이스 추가 |

**조건부 산출물:**

| 산출물 | 생성 조건 |
|--------|----------|
| 미팅 서머리 (고객용) | script 있고 summary 없는 회차 |
| 내부 정리 (아키텍트용) | script 있고 internal 없는 회차 |
| Phase 0 WHY | 스폰서/니즈 확인됨 |
| Phase 1 ASSESS | 인프라/데이터 확인됨 |
| Phase 2 GOVERN | 거버넌스 정보 확인됨 |
| ALFA 입력 프로필 (확정) | 6항목 대부분 🟢 |
| **유즈케이스 정의서** | 유즈케이스 식별됨 |
| **IRM 투입 판단** | 유즈케이스 확정됨 |

## IRM 스킬 (기술 설계)

### /irm-init — Phase 0 리소스 분류

| 항목 | 내용 |
|------|------|
| **언제** | 유즈케이스 확정 후 |
| **입력** | 유즈케이스 정의서 (자동 로드) + 프로필 (참조) |
| **출력** | `phase0_resource_inventory.md`, `phase0_classification.md`, `phase0_query_mapping.md` |
| **핵심** | Q1→Q2→Q3 분류 → 5가지 행위 유형 + 예상 질의 매핑 |
| **레퍼런스** | 실행가이드 v10 |

**Q1→Q2→Q3 분류:**
```
Q1. 위치를 아는가?
  아니오(내부) → 탐색(내부)
  아니오(외부) → 탐색(외부)
  예 → Q2. 절차가 있는가?
    아니오 → 접근
    예 → Q3. 결정적이고 재사용 가능?
      예 → 실행(확정)
      아니오 → 실행(유연)
```

### /irm-search — Phase 1 탐색 설계

| 항목 | 내용 |
|------|------|
| **언제** | Phase 0 후 |
| **입력** | Phase 0 결과 (자동 로드) |
| **출력** | `phase1_search.md` |
| **핵심** | 컬렉션 설계 + 임베딩/검색/리랭킹 + GraphRAG 판단 |
| **레퍼런스** | RAG 설계방법론 §1~§8, GraphRAG 설계방법론 §9~§12 |

**컬렉션 분리 3기준:** 갱신 주기 / 메타데이터 / 질의 패턴 → 2/3 이상 분리면 분리

**GraphRAG 4조건:** 용어 불일치 / 인과관계 >20% / RAG 적중률 미달 / 동일 엔티티 다중 맥락 → 2/4 이상 승인

### /irm-access — Phase 1 접근 설계

| 항목 | 내용 |
|------|------|
| **언제** | Phase 0 후 (/irm-search와 병렬 가능) |
| **입력** | Phase 0 결과 (자동 로드) |
| **출력** | `phase1_access.md` |
| **핵심** | 시스템별 CLI + REST 게이트웨이 설계 |
| **레퍼런스** | CLI 런북 v10 |

**CLI 원칙:** 시스템 단위 경계, JSON 표준 출력, 읽기전용 우선

**주의:** 현재 온프레미스(LLM과 레거시 같은 네트워크) 가정으로 설계됨. 클라우드 환경에서는 REST 게이트웨이 경유 구조 필요.

### /irm-exec — Phase 1 실행+체이닝 설계

| 항목 | 내용 |
|------|------|
| **언제** | Phase 0 후 (/irm-search, /irm-access와 병렬 가능) |
| **입력** | Phase 0 결과 (자동 로드) |
| **출력** | `phase1_execution.md`, `phase1_chaining.md` |
| **핵심** | 실행(확정) 파이프라인 + 실행(유연) LLM판단 + 체이닝 플로우 |
| **레퍼런스** | 실행가이드 v10 |

**실행(확정):** 같은 입력 → 같은 결과 (결정적)
**실행(유연):** LLM 판단 + Human-in-the-loop (보조수단)
**고정화:** 실행(유연)에서 패턴 안정 시 → 실행(확정) 전환

## 관통 원칙

| 원칙 | 설명 |
|------|------|
| **신뢰도 전파 (✅/⚠️)** | 리소스가 추정이면 분류도 추정, 설계도 추정 → 구현 전 검증 필수 |
| **내부/외부 용어 분리** | 고객용 문서에 ALFA/IRM/GraphRAG 절대 노출 금지 |
| **Intent → Behavior** | Phase 0에서 기술 결정 안 함. "인과관계 탐색 필요"로 분류 |

## 진짜도 스펙트럼

```
ALFA (미팅/프로필/유즈케이스)     → 진짜
IRM Phase 0 (리소스 분류)        → 반반 (실제 확인 + 추정 혼재)
IRM Phase 1 (설계 가설)          → 가설 (진짜 데이터 기반, 구현 후 검증)
IRM Phase 2 (구현 계획)          → 가설 기반 계획
```
