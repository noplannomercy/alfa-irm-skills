# Troubleshooting — FAQ & 문제 해결

---

## 순서/의존성 관련

### Q. 스킬 실행 순서를 꼭 지켜야 하나?

ALFA는 순서 필수, IRM은 일부 병렬 가능:

```
필수 순서:
  /alfa-prep → /alfa-live → /alfa-wrap  (순서 고정)
  /irm-init → (search, access, exec 병렬 가능) → /irm-design → /irm-validate

병렬 가능:
  /irm-search ─┐
  /irm-access ─┤→ /irm-design
  /irm-exec   ─┘
```

### Q. /alfa-live 없이 /alfa-wrap을 돌리면?

script 파일이 없으면 생성할 산출물이 없다. "먼저 /alfa-live를 실행하세요"라고 안내됨.

### Q. /irm-init 없이 /irm-search를 돌리면?

Phase 0 결과가 없으면 탐색 대상 리소스를 모른다. "/irm-init을 먼저 실행하세요"라고 안내됨.

### Q. /irm-design 없이 /irm-validate를 돌리면?

가능하긴 하지만 종합 요약 없이 체크리스트만 나온다. /irm-design을 먼저 돌리는 걸 권장.

## ALFA 관련

### Q. 미팅 1회만으로 유즈케이스가 나올 수 있나?

가능. 고객이 구체적인 요구("이 업무에 AI 쓰고 싶다")를 가지고 오면 1회면 충분. 반대로 "AI 도입하고 싶은데 뭘 해야 할지 모르겠다"면 2~3회 필요.

### Q. /alfa-live를 같은 미팅에서 여러 번 호출해도 되나?

된다. 같은 미팅 내 추가 메모면 같은 회차 script에 append되고, 프로필이 누적 업데이트됨.

### Q. /alfa-wrap을 미팅마다 돌려야 하나, 다 끝나고 한번에?

**미팅마다 돌리는 걸 권장.** 고객에게 서머리를 회신해야 하니까. wrap은 summary가 없는 회차에 대해서만 생성하므로, 여러 번 돌려도 중복 생성 안 됨.

### Q. 고객용 문서에 ALFA/IRM 용어가 나오면?

버그. summary에는 절대 나오면 안 됨. internal에만 프레임워크 용어 사용. 발견하면 수정 필요.

### Q. 프로필 빈칸이 많은데 wrap을 돌려도 되나?

된다. 있는 것만으로 산출물 생성하고, 부족한 부분은 명시. 유즈케이스 정의서나 IRM 투입 판단은 조건 미충족이면 생성 안 됨.

## IRM 관련

### Q. GraphRAG가 무조건 필요한 건가?

아니다. 4가지 조건 중 **2개 이상 충족**될 때만 승인:
1. 동음이의어/용어 불일치 빈번
2. 인과관계 질의 >20%
3. RAG 적중률 목표 미달
4. 동일 엔티티 다중 맥락

물류회사 테스트에서는 0/4로 불필요 판정.

### Q. 리소스가 전부 ⚠️추정이면?

설계는 가능하지만, **구현 전에 반드시 리소스 검증**이 필요. 추정 기반 설계가 실제와 다르면 IRM을 재실행해야 함.

### Q. 같은 리소스가 탐색이면서 접근일 수 있나?

있다. **같은 데이터라도 의도가 다르면 다른 행위 유형.** 예:
- MES 불량 이력 "현재 건 조회" → 접근 (위치 알고 직접 조회)
- MES 불량 이력 "유사 건 검색" → 탐색 (의미 기반 검색)

### Q. 실행(유연)이 실행(확정)으로 바뀌는 건 언제?

**고정화 조건**: 동일 입력 패턴에서 AI 판단과 사람 판단이 N회 연속 일치하면 규칙화 검토. 자동으로 바뀌는 게 아니라, 검토 후 결정.

### Q. 유즈케이스를 추가하면?

같은 고객 폴더에서 `/alfa-live`부터 다시:
1. 추가 미팅 → `/alfa-live` → `/alfa-wrap` (새 유즈케이스 정의서)
2. `/irm-init` (새 유즈케이스 폴더)
3. `/irm-search` ~ `/irm-validate`

**2번째 유즈케이스는 인프라/컬렉션 재사용으로 기간 대폭 절감.**

## 인프라 관련

### Q. 온프레미스가 아닌 클라우드 환경이면?

현재 `/irm-access`의 CLI 설계는 **LLM과 레거시가 같은 네트워크**라는 가정. 클라우드(Azure/AWS)에서 LLM을 운영하면 고객 사내 DB에 직접 접근 불가.

**대응:**
- REST 게이트웨이를 고객 사내에 설치
- LLM → 게이트웨이 → 레거시 DB 구조
- 또는 VPN/PrivateLink 연결

이 부분은 스킬 개선 예정.

### Q. GPU가 없는 고객이면?

클라우드 API(Bedrock, Azure OpenAI 등) 사용. 단, 데이터 외부 전송 가능 여부 확인 필수. 보안 요건으로 외부 전송 불가면 GPU 조달이 선행 조건.

## 미팅 메모 품질 관련

### Q. /alfa-live에 미팅 메모를 대충 넣으면?

프로필이 빈칸투성이가 되고, 신호 감지를 놓침. "같은 불량인데 다르게 기록한다" 같은 발언이 누락되면 GraphRAG 신호를 놓치고, IRM 설계가 완전히 달라질 수 있음.

**권장:** 미팅 녹취 → 텍스트 변환(Whisper, Clova Note 등) → 그대로 /alfa-live에 붙여넣기. 요약하지 말고 원문 그대로 넣는 게 안전함. 스킬이 알아서 추출함.

### Q. 미팅 메모에 뭘 꼭 포함해야 하나?

최소한:
- **참석자** (누가 왔는지 — 보안담당 동석 여부 등)
- **구체적 발언** ("같은 건데 다르게 불러요" 같은 원문)
- **숫자** (문서 몇 건, 이력 몇 건, 예산 얼마)
- **시스템명** (MES, SAP, CRM 등 구체적으로)
- **제약** ("데이터 외부 반출 안 됨" 같은)

## 파일/폴더 관련

### Q. 기존 고객 폴더가 있는데 다시 /alfa-prep을 돌리면?

기존 프로필(`alfa/profile.md`)을 로드하여 이어간다. 새로 만들지 않음.

### Q. STATUS.md 체크박스가 안 맞으면?

수동으로 수정해도 됨. STATUS.md는 진행 추적용이지, 스킬 동작에 영향을 주지 않음.

### Q. Phase 문서가 이미 있는데 wrap을 다시 돌리면?

덮어쓰지 않고 **보강**(Edit). 새로 확인된 정보가 있을 때만 업데이트. 버전이 올라감 (v1 → v2).
