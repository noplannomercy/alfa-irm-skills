# IRM Phase 0: 리소스 분류 및 질의 매핑

> 유즈케이스의 리소스를 파악하고, 예상 질의를 행위 유형으로 분류한다.

## 참조 문서

실행 시 아래 레퍼런스를 Read 도구로 읽어서 참조한다:
- `~/.claude/skills/irm-references/4.IRM_실행가이드_v10.md` — Phase 0 실행 절차, Q1/Q2/Q3 분류 기준, 행위 유형 정의

## 핵심 원칙

- **Intent → Behavior, Not Technology**: Phase 0에서는 기술을 결정하지 않는다. "GraphRAG가 필요하다"가 아니라 "인과관계 탐색이 필요하다"로 분류한다.
- 같은 데이터라도 의도가 다르면 다른 행위 유형이 된다.
- **신뢰도 전파**: 리소스의 근거(✅실제/⚠️추정)가 분류→매핑→설계→구현까지 전파된다. 추정 기반 항목은 구현 전 검증이 필수.

## 실행 흐름

### Step 0: 유즈케이스 정의서 로드

고객 폴더에서 유즈케이스 정의서를 찾는다:
- `*/alfa/usecase_*.md` 검색
- 여러 개면 AskUserQuestion으로 어떤 유즈케이스인지 확인
- 없으면 `/alfa-wrap`을 먼저 실행하라고 안내

유즈케이스명으로 `irm/[유즈케이스명]/` 폴더를 생성한다.

### Step 1: 리소스 인벤토리

유즈케이스 정의서 + 고객 프로필(`alfa/profile.md`)을 기반으로 관련 리소스를 도출한다.

AskUserQuestion으로 리소스 확인:
```
[유즈케이스명]에 관련된 리소스를 정리합니다.
아래 초안을 확인하고 빠진 것이나 수정할 것이 있으면 알려주세요.
(없으면 "확인"이라고 입력)
```

**파일:** `irm/[유즈케이스명]/phase0_resource_inventory.md`

```markdown
# [유즈케이스명] — 리소스 인벤토리

> 작성일: [날짜]
> 고객: [회사명]

## 리소스 목록

| # | 리소스명 | 유형 | 위치/접근 방법 | 근거 | 비고 |
|---|---------|------|--------------|:----:|------|
| 1 | [이름] | 문서/파일 | [경로/시스템] | ✅/⚠️ | [메모] |
| 2 | [이름] | 시스템 | [DB/API] | ✅/⚠️ | [메모] |
| 3 | [이름] | 절차/매뉴얼 | [위치] | ✅/⚠️ | [메모] |
| 4 | [이름] | 외부소스 | [URL/API] | ✅/⚠️ | [메모] |
| ... | ... | ... | ... | ... | ... |

### 근거 기준
- ✅ **실제**: IT팀 제공 데이터, 샘플 확인됨, 스키마/API 스펙 수령됨
- ⚠️ **추정**: 미팅 구두 정보, 유즈케이스에서 유추, 미확인

## 리소스 유형별 요약
- 문서/파일: [N]건 (✅[N] / ⚠️[N])
- 시스템: [N]건 (✅[N] / ⚠️[N])
- 절차/매뉴얼: [N]건 (✅[N] / ⚠️[N])
- 외부소스: [N]건 (✅[N] / ⚠️[N])
- **합계: [N]건 — 신뢰도: ✅[N]건 ([%]) / ⚠️[N]건 ([%])**
```

### Step 2: 리소스 분류 (Q1→Q2→Q3)

각 리소스에 대해 3가지 질문으로 행위 유형을 분류한다.

**분류 질문:**
```
Q1. 위치를 아는가? (데이터가 어디 있는지 알고 있는가?)
  - 아니오 (내부) → 탐색(내부)
  - 아니오 (외부) → 탐색(외부)
  - 예 → Q2로

Q2. 처리 절차가 있는가? (정해진 순서/규칙이 있는가?)
  - 아니오 → 여기서 끝 (탐색 또는 접근만)
  - 예 → Q3로

Q3. 절차가 결정적이고 재사용 가능한가? (같은 입력 → 같은 결과?)
  - 예 → 실행(확정)
  - 아니오 → 실행(유연)
```

**5가지 행위 유형:**

| 행위 유형 | 정의 | 예시 |
|----------|------|------|
| 탐색(내부) | 내부 문서에서 의미 기반 검색 | "유사 불량 이력?" |
| 탐색(외부) | 외부 정보 검색/크롤링 | "ISO 규정?" |
| 접근 | 위치 알고 있음, 직접 조회 | "현재 서버 CPU?" |
| 실행(확정) | 결정적, 재사용 가능 절차 | "검사 실행" → 같은 입력 = 같은 결과 |
| 실행(유연) | 상황 판단 필요한 절차 | "불량 원인 분석" → 케이스마다 다름 |

**파일:** `irm/[유즈케이스명]/phase0_classification.md`

```markdown
# [유즈케이스명] — 리소스 분류

> 작성일: [날짜]

## 분류 결과

| # | 리소스명 | Q1 위치 | Q2 절차 | Q3 결정적 | → 행위 유형 | 근거 | 비고 |
|---|---------|:-------:|:------:|:--------:|-----------|:----:|------|
| 1 | [이름] | Y/N | Y/N | Y/N | [유형] | ✅/⚠️ | [메모] |
| 2 | [이름] | Y/N | Y/N | Y/N | [유형] | ✅/⚠️ | [메모] |
| ... | ... | ... | ... | ... | ... | ... | ... |

**근거 전파 규칙:** 리소스 인벤토리의 근거(✅/⚠️)를 그대로 가져온다. 리소스가 ⚠️추정이면 분류 결과도 ⚠️추정.

## 행위 유형별 분포

| 행위 유형 | 리소스 수 | 비율 | 신뢰도 |
|----------|----------|------|:------:|
| 탐색(내부) | [N] | [%] | ✅[N]/⚠️[N] |
| 탐색(외부) | [N] | [%] | ✅[N]/⚠️[N] |
| 접근 | [N] | [%] | ✅[N]/⚠️[N] |
| 실행(확정) | [N] | [%] | ✅[N]/⚠️[N] |
| 실행(유연) | [N] | [%] | ✅[N]/⚠️[N] |

## 분포 해석
- 주요 유형: [탐색/접근/실행] 중심 ([%])
- 아키텍처 방향 시사: [한 줄 해석]
- **분포 신뢰도**: ✅확정 [%] / ⚠️추정 [%] — [추정 비율 높으면 "구현 전 리소스 검증 필수" 명시]
```

### Step 3: 질의 매핑

예상 질의 10~15개를 도출하고, 각 질의의 행위 유형과 관련 리소스를 매핑한다.

**파일:** `irm/[유즈케이스명]/phase0_query_mapping.md`

```markdown
# [유즈케이스명] — 질의 매핑 (IRM Mapping Table)

> 작성일: [날짜]

## 예상 질의 목록

| # | 예상 질의 | Q1 | Q2 | Q3 | → 행위 유형 | 관련 리소스 | 신뢰도 |
|---|----------|:--:|:--:|:--:|-----------|-----------|:------:|
| 1 | "[질의]" | Y/N | Y/N | Y/N | [유형] | #1, #3 | ✅/⚠️ |
| 2 | "[질의]" | Y/N | Y/N | Y/N | [유형] | #2 | ✅/⚠️ |
| ... | ... | ... | ... | ... | ... | ... | ... |

**신뢰도 판단 규칙:** 관련 리소스가 모두 ✅이면 ✅, 하나라도 ⚠️이면 ⚠️

## 질의별 행위 유형 분포

| 행위 유형 | 질의 수 | 비율 | 대표 질의 |
|----------|--------|------|----------|
| 탐색(내부) | [N]/[총] | [%] | "[대표]" |
| 탐색(외부) | [N]/[총] | [%] | "[대표]" |
| 접근 | [N]/[총] | [%] | "[대표]" |
| 실행(확정) | [N]/[총] | [%] | "[대표]" |
| 실행(유연) | [N]/[총] | [%] | "[대표]" |

## 체이닝 패턴 (복합 질의)

복수 행위 유형을 연결하는 질의:
- "[질의]" = 접근(시스템A) → 탐색(이력) → 실행(유연, 판단)
- ...

## Phase 0 결론

### 행위 유형 분포 요약
- **주요 유형**: [유형] ([%])
- **아키텍처 시사**: [탐색 중심 → RAG 투자 / 실행 중심 → CLI 파이프라인 / 혼합 → 체이닝]

### GraphRAG 사전 신호 (Phase 1에서 정식 판단)
- 동음이의어/용어 불일치: [있음/없음] — 예: [...]
- 인과관계 질의 비율: [N]/[총] ([%])
- 동일 엔티티 다중 맥락: [있음/없음] — 예: [...]
- → 사전 판단: [후보/불필요/미확인]

### 다음 단계
→ /irm-design로 Phase 1 (배치 설계) 진행
```

### Step 4: STATUS.md 업데이트

`STATUS.md`에 IRM 진행 상태를 추가/업데이트한다.

## 주의 사항

- Phase 0에서는 **기술 결정을 하지 않는다** (GraphRAG, 벡터DB 등은 Phase 1에서)
- GraphRAG 신호는 "사전 신호"로만 기록, 정식 판단은 `/irm-design`에서
- 리소스 수는 유즈케이스 범위에 따라 5~22개 (ALFA 스코핑된 유즈케이스는 보통 5~8개)
- 질의는 실제 사용자가 물어볼 법한 자연어로 작성
- 체이닝 패턴이 보이면 반드시 기록 (Phase 1 설계에 핵심)
