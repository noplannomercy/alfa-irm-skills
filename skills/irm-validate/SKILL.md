- `~/.claude/skills/irm-references/2.IRM_GraphRAG_설계방법론_v10.md` — §9~§12 구현 체크리스트 상세

### Step 1: 탐색 구현 체크리스트 — 탐색 유형이 있을 때만

**파일:** `irm/[유즈케이스명]/phase2_search_checklist.md`

```markdown
# [유즈케이스명] — 탐색 구현 체크리스트

> 작성일: [날짜]

## RAG 기본 (§1~§8)

### §1 컬렉션 설계
- [ ] 컬렉션 분리 기준 확정 (갱신주기/메타데이터/질의패턴)
- [ ] 컬렉션별 소스 매핑 완료
- [ ] 컬렉션명 확정

### §2 메타데이터 스키마
- [ ] 공통 필드 정의 (doc_type, dept_name, date, source)
- [ ] 도메인 필드 정의 (컬렉션별 2~4개)
- [ ] 추출 방식 결정 (자동/규칙/수동/소스시스템)

### §3 문서 파싱 & 청킹
- [ ] 파서 선택 (Docling 등)
- [ ] 청킹 전략 (HybridChunker — 구조 인식)
- [ ] 토크나이저 정렬 (모델 컨텍스트 길이)

### §4 임베딩 모델
- [ ] 모델 선정: [모델명]
- [ ] 로컬/API 결정
- [ ] 벡터 차원 확인
- [ ] 테스트 임베딩 품질 확인

### §5 검색 전략
- [ ] 벡터 단독 / 하이브리드 (벡터+BM25) 결정
- [ ] RRF 또는 가중치 조합 설정
- [ ] 크로스 컬렉션 라우팅 규칙

### §6 리랭킹
- [ ] 적용 여부 결정
- [ ] 리랭커 모델: [모델명]
- [ ] 적용 조건 (정밀도 낮음 / 다중 컬렉션 / false positive 높음)

### §7 인덱싱 범위
- [ ] 이력형: [범위] (예: 3년+, 핵심은 무제한)
- [ ] 최신형: 이전 버전 삭제 정책

### §8 갱신 파이프라인
- [ ] 즉시: [트리거 이벤트]
- [ ] 주기적: [스케줄]
- [ ] 수동: [조건]
- [ ] doc_id 매핑 (증분 업데이트용)

## GraphRAG 추가 (§9~§12) — 승인 시만

### §9 인프라
- [ ] Apache AGE 설치 (pgvector와 같은 PostgreSQL)
- [ ] 그래프 스키마 정의

### §10 엔티티/트리플 추출
- [ ] LLM 기반 추출 설정
- [ ] 제약조건: 자기참조 금지, 동사형 관계, 인과관계만
- [ ] 추출 품질 검증

### §11 그래프 검색 통합
- [ ] 멀티홉 경로 쿼리 (기본 2-hop)
- [ ] 컨텍스트 어셈블리 ([관련 문서] + [관련 관계])

### §12 갱신 동기화
- [ ] 벡터 + 그래프 동기 갱신
- [ ] 문서 삭제 시: 벡터 삭제 + 엣지 삭제 + 고아 정리
```

## 구현 범위 요약

| 구성요소 | 신규/재사용 | 우선순위 | 신뢰도 | 예상 기간 |

**구현 우선순위 규칙:**
- ✅확정 항목 먼저 착수 (바로 구현 가능)
- ⚠️추정 항목은 리소스 검증 완료 후 착수
- 추정 항목이 검증에서 달라지면 설계 재검토 (IRM 재실행 가능)

## 구현 순서 (의존성 기반)

```
Phase A: 인프라 ([N주])
  └─ [벡터DB, LLM, 기본 환경]

Phase B: 탐색 ([N주])
  └─ [컬렉션 구축, 파싱, 임베딩, 검색]
  └─ 의존: Phase A

Phase C: 접근 ([N주])
  └─ [CLI 구현, REST 게이트웨이]
  └─ 의존: Phase A

Phase D: 실행 ([N주])
  └─ [파이프라인, LLM 판단]
  └─ 의존: Phase B + C

Phase E: 통합 + 검증 ([N주])
  └─ [체이닝, 엔드투엔드, PoC]
  └─ 의존: Phase D
```

## 주의 사항

- Phase 2는 **설계 문서**이지 코드가 아님 — 실제 구현은 별도 작업
- 체크리스트 항목은 해당 유형이 있을 때만 생성 (빈 체크리스트 만들지 않음)
- 다른 유즈케이스와의 공유 가능성을 반드시 확인 (2번째 유즈케이스는 인프라/컬렉션 재사용으로 대폭 절감)
- 검증 기준은 정량적으로 (% , 초, 건수)
- PoC 범위는 고객 프로필의 제약사항(예산/일정/인력)을 반영
