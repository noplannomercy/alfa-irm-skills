# ALFA 미팅 후 산출물 생성기

> 미팅 완료 후 고객용/내부용 문서를 회차별로 생성한다.

## 폴더 구조

이 스킬은 기존 고객 폴더에 산출물을 추가한다.

```
[회사명]/
├── STATUS.md                    ← 업데이트
├── meeting/
│   ├── prep_질문리스트.md
│   ├── 1st_script.md            ← LIVE에서 생성됨
│   ├── 1st_summary.md           ← 이 스킬이 생성 (고객용)
│   ├── 1st_internal.md          ← 이 스킬이 생성 (내부용)
│   ├── 2nd_script.md
│   ├── 2nd_summary.md
│   ├── 2nd_internal.md
│   └── ...
├── alfa/
│   ├── profile.md               ← 기존 (참조)
│   ├── alfa_input_profile.md    ← 프로필 확정 시 생성
│   ├── phase0_why.md            ← Phase 0 확인 시 생성
│   ├── phase1_assess.md         ← Phase 1 확인 시 생성
│   ├── phase2_govern.md         ← Phase 2 확인 시 생성
│   └── governance_checklist.md  ← 거버넌스 Heavy 시
├── usecase/
│   ├── YYYYMMDD_[유즈케이스명].md  ← 유즈케이스 식별 시 (날짜 prefix)
│   └── ...                         ← 계속 추가 가능
└── irm/
    └── [유즈케이스명]/             ← 유즈케이스별 IRM (irm-init에서 생성)
        └── phase0_readiness.md    ← 투입 판단만 여기서 생성
```

## 실행 흐름

### Step 1: 고객 폴더 탐색 및 상태 확인

기존 고객 폴더를 찾는다:
- `*/STATUS.md` 파일을 Glob으로 검색
- 여러 개면 AskUserQuestion으로 어떤 고객인지 확인
- 없으면 `/alfa-prep` → `/alfa-live` 순서로 먼저 실행하라고 안내

`STATUS.md`, `alfa/profile.md`, `meeting/*_script.md`를 모두 로드하여:
- 현재 미팅 회차 확인
- 프로필 완성도 확인
- 아직 summary/internal이 없는 미팅 회차 확인 → 이번에 생성할 대상
- 기존 Phase 문서 존재 여부 확인

### Step 2: 산출물 생성 범위 결정

**회차별 산출물:**
- summary/internal이 없는 모든 미팅 회차에 대해 생성

**프로필 완성도 기반 추가 산출물:**

| 조건 | 생성 문서 | 저장 위치 |
|------|----------|----------|
| 항목 1,5 확인됨 | 미팅 서머리 + 내부 정리 | `meeting/` |
| Phase 0 정보 확인됨 | Phase 0 WHY 문서 | `alfa/` |
| Phase 1 정보 확인됨 | Phase 1 ASSESS 문서 | `alfa/` |
| Phase 2 정보 확인됨 | Phase 2 GOVERN 문서 | `alfa/` |
| 항목 1~6 대부분 확인됨 | ALFA 입력 문서 (프로필 확정) | `alfa/` |
| 유즈케이스 식별됨 | 유즈케이스 정의서 초안 | `usecase/` |
| 유즈케이스 확정됨 | IRM Phase 0 투입 판단 | `irm/` |

**Phase 문서 생성 규칙:**
- 이미 존재하는 Phase 문서는 **덮어쓰지 않고 보강** (Edit)
- 새로 확인된 정보가 있을 때만 업데이트
- Phase 0, 1, 2는 최초 한번 생성 후 확정

### Step 3: 문서 생성

#### 3-1. 미팅 서머리 (고객용) — 회차별

**파일:** `[회사명]/meeting/[N]st_summary.md`

```markdown
# [회사명] [N]차 미팅 서머리

> 일시: [날짜]
> 참석: [참석자]

## 논의 내용
1. [주제별 요약 — 비기술적, 의사결정 중심]
2. ...

## 합의 사항
- [합의 1]
- [합의 2]

## 다음 스텝
| 항목 | 담당 | 기한 |
|------|------|------|
| [내용] | [누구] | [언제] |

## 참고
- [추가 전달 사항]
```

**톤 규칙:**
- ALFA, IRM, GraphRAG, 거버넌스 무게 등 내부 용어 **절대 사용 금지**
- 고객이 읽어도 자연스러운 비즈니스 언어로 작성
- "AI 도입 로드맵", "파일럿 프로젝트", "내부 검토" 등의 일반 용어 사용

**로드맵 유형 선택 기준:**

```
경량 (4주) — 조건: 클라우드 OK + 규제 Light + 내부용
  Week 1: Bedrock 개통 + 벡터DB + 문서 파싱
  Week 2: RAG 파이프라인 + 프로토타입
  Week 3-4: 피드백 + 개선 + 확장 판단

표준 (3개월) — 조건: 온프레미스 + 규제 Light
  Week 1-4: GPU/Ollama 세팅 + 벡터DB + 문서 준비
  Week 5-8: RAG 파이프라인 + 검색 서비스
  Week 9-12: 파일럿 + GraphRAG 검토 + 확대

규제 Heavy (3~4개월) — 조건: 금융/의료/공공
  Week 1-2: 거버넌스 확정 + Bedrock 개통
  Week 3-4: 인프라 세팅 (Bedrock, 벡터DB, 가드레일)
  Week 5-10: 파일럿 (2주 운영 포함)
  Week 11-14: 평가 + 전사 확대
```

#### 3-3. ALFA Phase 문서 — 점진적 생성 및 보강

Phase 문서는 미팅 회차마다 점진적으로 채워진다. 정보가 충분한 Phase만 생성하고, 이미 존재하면 보강한다.

**생성/보강 규칙:**
- 문서가 없고 해당 Phase 정보가 확인됨 → 새로 생성 (상태: 초안 또는 확정)
- 문서가 있고 새 정보가 추가됨 → 보강 (버전 올림, 변경 이력 추가)
- 문서가 있고 변경 없음 → 건드리지 않음
- 상태: `초안` (빈칸 있음) → `부분` (대부분 채움) → `확정` (완성)
- 유즈케이스 추가나 컴플라이언스 변경으로도 보강 가능

### Step 4: STATUS.md 최종 업데이트

`STATUS.md`를 업데이트한다:

## 주의 사항

- 고객용 서머리에는 ALFA/IRM/GraphRAG/거버넌스 무게 등 내부 용어 **절대 금지**
- 내부 정리에는 프레임워크 용어 적극 사용 (팀 소통용)
- 프로필 빈칸이 많아도 있는 것만으로 산출물 생성 — 부족한 부분은 명시
- IRM 투입 판단은 유즈케이스가 확정되었을 때만 (섣불리 판단하지 않음)
- 로드맵은 프로필 기반으로 **가장 적합한 유형**을 선택하되, 변동 가능성 명시
- 회차별 summary/internal은 해당 회차의 script만 기반으로 생성 (이전 회차 내용 반복 금지)
- Phase 문서는 최초 1회 생성 후 확정 — 이미 존재하면 보강만 (덮어쓰지 않음)
- 유즈케이스 파일명에 날짜 prefix 필수 (YYYYMMDD)
- usecase/ 폴더는 유즈케이스가 식별되었을 때만 생성
- irm/ 하위 폴더는 유즈케이스명으로 생성
