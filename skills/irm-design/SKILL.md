# IRM Phase 1: 배치 설계 (Deployment Design)

> Phase 0의 행위 유형 분포를 기반으로, 각 유형별 기술 배치를 설계한다.

## 핵심 원칙

- **단계적 성숙**: 단순 → 복잡 (RAG → GraphRAG, Prompt → CLI)
- **증거 기반 승격**: GraphRAG는 조건 2/4 이상일 때만
- **단일 인터페이스, 다중 구현**: 외부 인터페이스는 불변, 내부만 진화
- **신뢰도 전파**: Phase 0의 ✅/⚠️를 이어받는다. ⚠️추정 기반 설계는 "검증 후 확정" 태그를 붙인다.

## 폴더 구조

```
[회사명]/
├── irm/
│   └── [유즈케이스명]/
│       ├── phase0_*.md                     ← 이미 존재
│       ├── phase1_search.md                ← 탐색 배치 (해당 시)
│       ├── phase1_access.md                ← 접근 배치 (해당 시)
│       ├── phase1_execution.md             ← 실행 배치 (해당 시)
│       ├── phase1_chaining.md              ← 체이닝 설계 (해당 시)
│       └── phase1_summary.md               ← 종합 요약
└── STATUS.md                               ← 업데이트
```

## 실행 흐름

### Step 0: Phase 0 결과 로드

고객 폴더에서 Phase 0 결과를 로드한다:
- `irm/[유즈케이스명]/phase0_*.md` 3개 파일 로드
- 없으면 `/irm-init`을 먼저 실행하라고 안내
- 행위 유형 분포를 확인하고, 해당하는 유형만 설계한다

### Step 6: STATUS.md 업데이트

## 주의 사항

- 다른 유즈케이스와 인프라/컬렉션 공유 가능 여부를 반드시 확인 (같은 고객의 2번째 유즈케이스는 대폭 절감 가능)
- GraphRAG는 증거 기반으로만 승인 (2/4 조건 이상)
- 실행(유연)은 초기에 Prompt 기반, 패턴 안정 시 CLI Agent로 전환
- 기술 선택은 고객 인프라 제약을 반영 (온프레미스 → 로컬 모델, 클라우드 → Bedrock/API)
- 모든 설계는 "외부 인터페이스 불변, 내부 구현 진화" 원칙 준수
